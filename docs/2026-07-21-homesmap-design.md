# 홈즈맵 (HOMES MAP) — 웹앱 상세 설계

> 우리집 물건 지도 · 가족이 함께 쓰는 물건 위치 관리 서비스

## 1. 개요

### 무엇인가
집 안 어디에 무엇이 있는지를 **집 지도 위에** 기록해두고, "어딨지?"를 검색 한 번으로 해결하는 가족 공유 서비스.

- 방을 그리고 → 방 안에 수납장(서랍장·옷장·냉장고 등)을 놓고 → 수납장에 물건을 등록
- "손톱깎이" 검색 → 어느 방 어느 수납장인지 지도에서 반짝이며 안내
- 가족 누구든 등록/검색, "누가 무엇을 넣었다" 활동 피드로 공유
- **소모품**(건전지·세제·참기름 등)은 재고 수량을 관리해서, 떨어지면 가족 공유 장보기 리스트에 자동으로 올리고 외부 쇼핑몰 구매 링크로 바로 연결

### 해결하는 문제
"그거 어디 뒀더라?"로 온 가족이 서랍을 다 뒤지는 시간 낭비 + "이거 다 떨어진 줄 몰랐네"로 인한 재구매 누락.

## 2. 서비스 성격 → 아키텍처 원칙

이 서비스의 본질이 아키텍처를 결정한다.

| 성격 | 설계 함의 |
|---|---|
| **데이터가 아주 작다** (가족당 방 몇·수납장 수십·물건 수백) | 전체를 기기에 통째로 캐싱 가능 → 서버 조각 요청 불필요 |
| **읽기 위주** (일상의 99%는 "어딨지?" 검색) | 조회는 서버 왕복 없이 로컬에서 즉시 |
| **폰 들고 그 자리에서** (서랍 앞, wifi 약한 곳) | **오프라인·즉각성이 핵심** → 로컬-퍼스트 + PWA |
| **동시 편집 거의 없음** (가족 2~5명) | 단순 동기화(LWW)로 충분, CRDT 불필요 |
| **매우 사적·민감** (귀중품 위치 지도) | **프라이버시가 서비스의 신뢰 그 자체** → 민감 항목 종단간(E2E) 암호화 |

**핵심 원칙 3가지**
1. **로컬-퍼스트**: 기기(IndexedDB)가 1차 데이터 소스. 조회·검색은 로컬에서 즉시, 오프라인 동작.
2. **서버는 동기화·백업·공유 계층**: Supabase는 인증 + 암호화된 데이터의 동기화 저장소 + 사진 저장.
3. **민감 데이터 E2E 암호화**: 물건 이름·메모·사진은 기기에서 암호화되어 서버엔 암호블롭만. 운영자도 못 읽음.

> 반영 전(rev1)은 SSR 서버 중심 SaaS였으나, 위 성격에 맞춰 로컬-퍼스트 + E2E로 전환. **사업 모델(여러 가족 가입 SaaS)은 동일.**

## 3. 범위와 단계 (Phasing)

한 번에 다 만들지 않는다. 단, **로컬-퍼스트 동기화 + E2E 암호화는 나중에 끼울 수 없는 근간**이라 v1의 토대로 포함한다.

| 단계 | 내용 |
|---|---|
| **v1 — 핵심** | 인증 + 가족/초대 + **E2E 키 관리** + **로컬-퍼스트 동기화(PWA)** + 지도 에디터 + 물건 등록 + 검색 + 활동 피드 |
| **v2 — 소모품** | 재고 수량 + 부족 시 장보기 리스트 자동 등록 + 외부 쇼핑몰 링크 |
| **v3 — 다듬기** | 실시간 반영(동기화 즉시성↑), 오프라인 마감, 성능, 모바일 방 그리기 UX |
| _future_ | AI 이미지 인식(사진→물건 자동 인식) — E2E와의 관계는 §11 |

## 4. 아키텍처

**Next.js(App Router, 주로 클라이언트) PWA + Supabase(동기화 백엔드) + Vercel.** 새 무거운 의존성 없음.

```
┌──────────────────────────────────────────────────────┐
│  기기 (PWA, 설치 가능)                                  │
│   ┌────────────────────────────────────────────────┐  │
│   │ UI (지도/검색/등록)  ← 로컬에서 즉시 렌더         │  │
│   │        ▲  ▼ (복호화된 데이터, 메모리에만)          │  │
│   │ 암호화 계층 (Web Crypto, 가족키로 enc/dec)        │  │
│   │        ▲  ▼                                       │  │
│   │ IndexedDB (암호블롭 + 구조 데이터 캐시) = 1차 소스 │  │
│   └───────────────────────┬────────────────────────┘  │
│         동기화 (pull/push, 오프라인 시 큐잉)             │
└───────────────────────────┼──────────────────────────┘
                            │ HTTPS
┌───────────────────────────▼──────────────────────────┐
│  Supabase (동기화·백업·공유 · 평문 못 읽음)             │
│   - Auth (카카오/구글/이메일)                          │
│   - Postgres + RLS: 암호블롭 + 구조 데이터 저장         │
│   - Storage: 암호화된 사진 바이트                      │
│   배포: Vercel (Root Directory = web/)                 │
└──────────────────────────────────────────────────────┘
```

- **평문이 기기를 떠나지 않음**(사진·물건 이름·메모). 서버는 복호화 키가 없음.
- 구조 데이터(방·수납장 위치/색/종류, 좌표)는 평문 — 비밀이 아니고 지도 렌더에 필요. **비밀은 "그 안에 무엇이 있는가"**.

## 5. 데이터 모델

### 5.1 저장 계층 두 곳
- **IndexedDB(기기)**: 서버 테이블의 미러 + 동기화 메타(`dirty`, `lastSync`). 앱은 여기서 읽는다.
- **Postgres(서버)**: 아래 테이블. 민감 컬럼은 암호블롭.

### 5.2 테이블 (rev1 대비 변경점 굵게)

```mermaid

    families ||--o{ family_members : "구성"
    families ||--o{ family_invites : "초대"

    rooms ||--o{ storages : "포함"
    storages ||--o{ items : "담음"
    families ||--o{ activity : "기록"
    families ||--|| family_keys : "래핑된 가족키"
```

### 5.3 설계 결정
- **암호화 대상 = 물건 `enc_name`·`enc_memo`·사진, 활동 `enc_payload`.** (활동 문구에 물건 이름이 들어가므로 함께 암호화)
- **평문 유지 = 방·수납장 이름/좌표/종류, 이모지, 물건-수납장 관계.** 지도 구조는 비밀이 아니고 렌더에 필요. (수납장 이름 "금고"처럼 민감할 수 있는 경계는 §13 결정사항)
- **동기화 컬럼**: 모든 동기화 테이블에 `updated_at`(LWW 기준) + `deleted_at`(소프트 삭제 → 삭제도 동기화됨). 물리 삭제는 주기적 정리로.
- **`family_id` 비정규화 유지**: RLS 한 홉. RLS는 E2E와 **다층 방어**로 병행(암호화 실패해도 남의 가족 블롭 접근 차단).
- 암호블롭 형식: `{ "iv": base64, "ct": base64 }` (AES-GCM, per-row 랜덤 IV).

## 6. 로컬-퍼스트 동기화

작은 데이터 + 낮은 동시성 → **CRDT 없이 last-write-wins(LWW)** 로 충분.

- **읽기**: 앱은 항상 IndexedDB에서 렌더. 서버 왕복 없음 → 즉시·오프라인.
- **동기화(pull)**: 진입/포커스 시 `select * where family_id=? and updated_at > lastSync`. 데이터가 작아 필요하면 전량 pull도 무방. 받은 행을 IndexedDB에 병합(LWW: `updated_at` 큰 쪽 승).
- **쓰기(push)**: 로컬 즉시 반영(낙관적) + `dirty` 표시 → 서버 upsert. 오프라인이면 큐에 쌓였다가 온라인 시 flush.
- **충돌**: 같은 행 동시 수정 시 `updated_at` 최신 승(=LWW). 가족 규모에서 실질 충돌 희박. `ponytail:` LWW로 시작, 실제 충돌 민원 생기면 필드 병합 고려.
- **삭제**: `deleted_at` 세팅 → 동기화로 전파 → 로컬에서 숨김.

**PWA**: `manifest.json`(홈 화면 설치) + 서비스워커(앱 셸 캐시로 오프라인 실행). 오프라인 **데이터**는 IndexedDB가, 오프라인 **앱 구동**은 SW가 담당.

## 7. E2E 암호화 & 키 관리 ★핵심

서버가 절대 복호화 키를 갖지 않으면서, 가족이 여러 기기에서 공유하는 방법.

### 7.1 키 구조
- **FDK(Family Data Key)**: 가족당 하나의 랜덤 256-bit 대칭키(AES-GCM). 실제 물건 이름/메모/사진을 이 키로 암·복호화.
- FDK는 **가족 생성 시 생성자 기기에서 랜덤 생성**되고, 서버로 평문 전송되지 않는다.

### 7.2 기기 간 유지 (같은 사람, 여러 기기)
- 사용자가 **홈즈맵 잠금 암호(패스프레이즈)** 설정 → `KDF(패스프레이즈)`로 래핑키 도출(Argon2id/PBKDF2).
- FDK를 래핑키로 암호화한 `wrapped_family_key`를 `family_members`에 저장.
- 새 기기: 로그인 후 패스프레이즈 입력 → 서버의 `wrapped_family_key` 받아 FDK 복원. **서버는 래핑블롭만 봄.**
- (OAuth는 서버가 못 보는 안정적 비밀을 안 주므로, 진짜 zero-knowledge엔 패스프레이즈가 불가피 — 이건 E2E의 본질적 비용.)

### 7.3 새 멤버에게 공유 (초대)
- 초대 링크에 FDK를 **URL 프래그먼트**로 실어 전달: `https://…/invite/<token>#k=<base64 FDK>`.
- **프래그먼트(`#…`)는 브라우저가 서버로 전송하지 않음** → FDK가 서버를 안 거침.
- 초대받은 사람 기기: 프래그먼트에서 FDK 취득 → 자기 패스프레이즈로 다시 래핑해 자기 `wrapped_family_key` 저장.
- 트레이드오프: **링크를 가진 사람이 곧 키를 가짐** → 초대 링크는 카톡 등 신뢰 채널로 가족에게만. (PKI/공개키 래핑은 4인 가족앱엔 과설계 → 채택 안 함.)

### 7.4 복구 (중요·본질적 트레이드오프)
- 패스프레이즈를 잊으면 그 기기에서 복호화 불가. → **다른 가족원의 새 초대 링크로 재참여**(그 사람은 FDK 보유)하면 복구.
- **모든 멤버가 접근을 잃으면(전원 패스프레이즈 분실 + 유효 초대 없음) 암호화 데이터는 영구 복구 불가.** 이것이 E2E의 본질.
- 완화책: ① 최소 2인 유지 권장, ② 가족 생성 시 **복구 코드(원본 FDK)를 1회 표시**해 안전한 곳에 보관하도록 유도(선택). §13에서 수준 결정.

### 7.5 암·복호화 흐름
- 등록: 물건 이름/메모 → FDK로 AES-GCM 암호화 → `{iv,ct}` 서버 저장. 사진 → 바이트 암호화 후 Storage 업로드.
- 조회: 동기화로 받은 암호블롭 → 기기에서 FDK로 복호화 → 메모리에서 렌더·검색. 평문은 디스크에 안 남김(IndexedDB엔 암호블롭 저장, 세션 중 복호화 캐시는 메모리).

## 8. 인증 & 멀티테넌시

### 8.1 인증
Supabase Auth. 카카오·구글·이메일 매직링크. (한국 가족 대상 → 카카오 1순위)

### 8.2 가족(테넌트) 모델
- 첫 진입: **새 가족 만들기**(→ owner, FDK 생성 + 패스프레이즈 설정) 또는 **초대 링크로 참여**(FDK 프래그먼트 취득 + 패스프레이즈 설정).
- 스키마는 유저↔가족 다대다 지원, **v1 UI는 유저당 활성 가족 1개**.

### 8.3 RLS
모든 테이블 `family_id`, 정책 = "요청 유저가 그 가족의 `family_members`에 존재". E2E와 **다층 방어**로 병행.

## 9. v1 기능 상세

- **지도 에디터**: 프로토타입 3-모드(둘러보기/방 그리기/수납장 놓기) 이식. 데스크톱=드래그 그리기, 모바일=탭. 구조 데이터는 평문.
- **물건 등록 (draft→확인→저장)**: 이름·메모·사진 입력 → **암호화** → 로컬 반영 + 서버 push + 활동 기록. draft 배열 추상화로 future AI 인식이 앞단에 꽂힘(§11). v1은 draft 1건.
- **검색**: 동기화·복호화된 로컬 데이터에서 클라이언트 필터(이름·메모 부분일치 상위 8). 결과 클릭 → 지도 방 glow + 수납장 pulse.
- **활동 피드**: `enc_payload` 복호화 후 프로토타입식 "이모지+누가+무엇" 렌더. 최근 50건.

## 10. v2 — 소모품 & 구매 연동
(rev1과 동일 방향, 저장 계층만 위 구조 따름)
- 물건에 "소모품" 토글 → `quantity`/`low_threshold`/`shopping_url`. (수량·임계는 민감도 낮아 평문 가능, `shopping_url`도 평문. 물건 이름은 여전히 암호화.)
- 재고 ≤ 임계 → 가족 공유 `shopping_list` 자동 등록(중복 방지, 재입고 시 내려감). 판정은 앱 로직 우선.
- 장보기 리스트 화면 + 항목별 "사러 가기"(외부 쇼핑몰 새 탭). 앱 내 결제·주기예측·가계부는 비범위.

## 11. future — AI 이미지 인식 (지금 미구현) + E2E 주의

- 기능 본질은 여전히 "사진 → 물건 후보 → 확인 → 등록"이라 §9 등록의 앞단에 끼움. 지금 인프라 추가 없음.
- **E2E와의 긴장**: 사진이 암호화되면 서버(Edge Function)가 못 읽음. 해법 — 인식은 **사용자가 명시적으로 시작**해 기기에서 복호화된 이미지를 그 순간에만(저장 없이) AI에 TLS로 전달. "서버는 저장 평문을 못 본다"는 유지하되, AI 사용은 사용자 동의 예외로 명시.

## 13. 열린 질문 / 결정 필요 (기본값 제안)

1. **복구 수준** — E2E 특성상 전원 접근 상실 시 데이터 소실. ① 복구 코드 1회 표시(선택 보관)만 제공 / ② 서버 보조 복구(프라이버시↓) 중? _(제안: ①, 최소 2인 유지 권장 + 복구 코드)_
2. **암호화 경계** — 수납장 이름("금고" 등)도 암호화? _(제안: v1은 물건 이름·메모·사진만, 수납장/방 이름은 평문. 필요 시 확장)_
3. **패스프레이즈 UX** — 매 기기 1회 입력 + 세션 유지 방식? 생체인증(WebAuthn)으로 잠금 해제는 나중? _(제안: v1은 패스프레이즈 1회 + 로컬 세션 유지, 생체는 v3)_
4. **KDF** — Argon2id(강함, wasm 의존) vs PBKDF2(WebCrypto 내장, 의존 없음)? _(제안: v1은 PBKDF2 고반복(내장, 의존 0), 필요 시 Argon2id)_
5. **초대 방식** — 링크(프래그먼트 키 포함)만? _(제안: 예. 6자리 코드는 키 전달이 안 돼 E2E와 안 맞음 → 링크 방식)_
6. **PWA 서비스워커** — 직접 최소 구현 vs `next-pwa` 의존? _(제안: 앱 셸만 캐시하는 최소 SW 직접 구현, 의존 0)_
7. **배포** — Root Directory=`web/`? _(제안: 예, 기존 동일)_
